# DevTools Noise Requests

## Purpose

Make DevTools Sandbox tasks harder by sending extra garbage requests when the intern presses the request button.

## Scope

- DevTools request button sends one real request and several decoy requests.
- Decoy requests use similar but distinct endpoint names and appear in browser Network.
- The correct response is still returned only for the configured endpoint and query.
- Decoy responses are generated by the internal DevTools sandbox route.

## Data Model

- No schema changes.
- Existing `Question.apiConfig.query`, `successBody`, `successStatus`, and `successHeaders` are reused.

## Main Flow

- Intern opens a `DEVTOOLS_SANDBOX` question.
- Intern clicks the DevTools request button.
- Client sends a shuffled batch: the configured request plus eight decoy requests.
- Decoy endpoint names are derived from the configured endpoint, for example `request_method_metrics`.
- Server returns configured `successBody` for exact endpoint/query match.
- Server returns plausible garbage JSON for wrong endpoint or query.

## Touched Files

- `components/intern/test-runner.tsx`
- `app/api/devtools-sandbox/[questionId]/[...endpoint]/route.ts`
- `docs/features/devtools-noise-requests/README.md`
- `docs/features/devtools-noise-requests/VERIFY.md`

## Constraints

- Decoy requests must not change stored answers or scoring.
- DevTools UI must still hide correctness during the test.
- The configured request must remain in the batch every time.
