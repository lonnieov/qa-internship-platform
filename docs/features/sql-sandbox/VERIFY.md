# SQL Sandbox Verification

## Preconditions

- локально применена Prisma schema с новым `QuestionType.SQL_SANDBOX`
- проект поднят на локальной базе
- есть доступ в админку

## Smoke Check

1. Открыть `/admin/questions`.
2. Переключиться на чипсу `SQL Sandbox`.
3. Убедиться, что видны форма нового вопроса и список SQL-задач.
4. На чистой базе проверить, что среди сидовых SQL-вопросов есть задачи:
   - про студентов, изучающих `Математику`
   - про покупки клиента `ООО Стройка`
   - про сотрудников отдела `ИТ`

Ожидаемый результат:
- экран не падает
- тип вопроса в форме не требует дополнительного ручного переключения
- сидовый SQL-вопрос отображается в списке без ручного добавления

## Positive Case

1. Войти как стажёр и начать ассессмент.
2. Дойти до SQL-вопроса про студентов и предмет `Математика`.
3. Ввести корректный запрос и нажать `Выполнить`.

Пример запроса:
```sql
select s.name
from Student s
join StudentSubject ss on s.id = ss.student_id
join Subject sub on ss.subject_id = sub.id
where sub.title = 'Математика'
order by s.name;
```

Ожидаемый результат:
- в верхней части редактора нет шильдиков с ожидаемыми колонками результата
- справа видна diagram-like панель таблиц со связями между сущностями
- область диаграммы можно двигать мышью и масштабировать через контролы
- внутри каждой таблицы на диаграмме сразу видны sample data
- снизу появляется таблица результата
- вопрос помечается как отвеченный
- в админском отчёте виден сам SQL-запрос и краткий итог

4. После корректного запуска ввести заведомо неверный запрос и снова нажать `Выполнить`.

Ожидаемый результат:
- стажёр видит ошибочный результат текущего запуска
- в админском отчёте по завершённой попытке остаётся сохранён последний корректный SQL-ответ, а вопрос не теряет статус `верно`

5. Дойти до SQL-вопроса про сотрудников отдела `ИТ`.
6. Ввести корректный запрос и нажать `Выполнить`.

Пример запроса:
```sql
select e.name
from Employee e
join Department d on e.department_id = d.id
where d.name = 'ИТ'
order by e.name;
```

Ожидаемый результат:
- появляется таблица результата с тремя строками:
  - `Алексеев Алексей`
  - `Волков Дмитрий`
  - `Егорова Анна`
- вопрос получает статус `зачтено`

## Negative Cases

1. Ввести `update Employee set full_name = 'Test';`

Ожидаемый результат:
- показывается ошибка, что разрешены только `SELECT` и `WITH`

2. Ввести два запроса через `;`

Ожидаемый результат:
- показывается ошибка, что разрешён только один SQL-запрос

3. Ввести синтаксически неверный SQL

Ожидаемый результат:
- показывается текст SQL-ошибки
- приложение не падает

4. Ввести запрос к таблице, которой нет в задании

Пример:
```sql
select * from Orders;
```

Ожидаемый результат:
- показывается понятная ошибка, что таблицы нет в задании
- запрос не выполняется

## Regression Check

1. Переключиться обратно на `Quiz`, `API Sandbox`, `DevTools Sandbox`, `Manual QA`.
2. Убедиться, что форма создания и список вопросов по-прежнему работают.
3. Запустить обычный quiz и API sandbox вопрос.

Ожидаемый результат:
- существующие типы вопросов не сломаны
- навигация по чипсам и порядок вопросов в тесте сохраняются
